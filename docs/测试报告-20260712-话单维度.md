> 日期：2026-07-12

## 一、发现的 Bug

### 🔴 [P0] 跨天查询 total 严重虚高（分页错乱）

**现象**
- 跨天无筛选 `07-06~07-07`：API 返回 `total:2152`，数据库实际仅 **14 行**
- 跨天含空表 `06~13`：API 返回 `total:2356`，实际 **15 行**
- 前端表现：分页器显示"共 2152 条 / 2356 条"，但每页只渲染真实数据（14/15 行），翻页逻辑完全错乱

**根因**
`handleList` 跨天/带筛选场景用 `EXPLAIN SELECT 1 FROM (UNION ALL 派生表) t` 估算 total。MySQL 对"UNION ALL 派生表 + 多个 LEFT JOIN + COLLATE 比较"的 `rows` 估算严重失真：
- 手动验证：跨天无筛选 EXPLAIN 派生表 rows 合计 **394**（实际 14）
- 上一轮修复只覆盖了"估算值**虚低**（小表误报 rows=1）"的 fallback，没覆盖"**虚高**"，导致该 bug 被误判为"正常的大数据量估算"漏测

**为什么上一轮没测出来（复盘）**
1. 上一轮冒烟测试里 CDR 列表查的是"今天"，今天空表走空分支，没触发
2. 上一轮只验证了"能返回 JSON + approx=true"，没做 **total 与库 COUNT 的交叉验证**——而 approx=true 恰好让我放松了警惕，把 1748/2152 当成了合理的大数据量
3. EXPLAIN 估算失真这种问题纯静态审查看不出，必须动态跑多维度 + 交叉验证

**修复**
删除 EXPLAIN 估算路径，跨天和带筛选一律 **精确 COUNT(*)**（逐表 COUNT 再 SUM，多表重复 params）；单天无筛选保留 `cdr_counter` 快路径；`approx` 恒为 `false`。

## 三、观察项（非 bug，但值得记录）

1. **CDR 数据 node_id 全为 0**：代码 `if (!empty($_GET['node_id']))` 中 PHP `empty('0') === true`，会忽略 `node_id=0` 筛选。但真实节点 ID 从 1 起，当前不影响查询。CDR 接收时未正确关联节点是**数据侧问题**，建议另立 issue 排查 cdr-receiver 的 matchNodeId 逻辑。
2. **上一轮修复遗留的 approx 标志**：现改为恒 `false`，前端 `pagination.approx` 不再有"估算"语义，可保留兼容或后续移除。
