# VOS3000 上线手册 — v17（2026-07-13）

> 适用：从旧版本**升级**到 v17，或**全新部署**后首次上线。
> 本版核心变更：CDR 通话记录**无筛选任意页码毫秒级分页**（counter 分段定位），真实 total 展示，分页按总条数铺开。
> 完整环境准备见 `docs/线上部署指南.md`（MySQL / PHP 扩展 / PM2 / Nginx / 防火墙）。

---

## ⚠️ v17 上线必读：一个必须先做的步骤（与 v16 相同）

v17 的「通话记录」列表**无筛选时直接用 `y_cdr_counter` 做两件事**：①算真实总数（秒级）；②把全局页码换算成「具体某天 + 当天局部偏移」做分段定位。

如果计数器与日表对不上，会出现「快但错」——显示几千条实际只有几条，或翻到某页数据错位。

我们本地实测就发现 counter 严重漂移（日表 1 行 counter 记 3008，有数据的日子 counter 却是 0）。**生产库极大概率同样漂移**（历史清过日表但 counter 没同步清）。

所以上线 v17 **必须执行一次计数器重建**：

```bash
cd /www/wwwroot/yump
php server/database/fix_cdr_counter.php
```

> 该脚本先清空 counter 再按日表名日期重新聚合，与 cdr-receiver 增量累加口径一致。幂等、不丢数据。
> 之后 counter 由 cdr-receiver 在有话单时自动维护，`handleClear` 清日表也会同步清 counter，无需再手动跑。

## 一、升级流程（生产已有旧版本）

```bash
# 0. 登录服务器，进入项目根目录
cd /www/wwwroot/yump

# 1. 备份配置（打包已排除 .env，但保险起见）
cp server/.env /tmp/yump.env.bak
cp /etc/yump/.env.node /tmp/yump.env.node.bak

# 2. 备份当前代码（用于回滚）
cp -r dist /tmp/yump-dist-bak
cp -r server /tmp/yump-server-bak

# 3. 上传 v17 包并解压（覆盖 dist/ 和 server/）
#    scp vos3000-deploy-20260713-v17.tar.gz 服务器:/www/wwwroot/yump/
tar -xzf vos3000-deploy-20260713-v17.tar.gz

# 4. 确认 .env 未被覆盖（应为备份时的值，未被改动）
grep YUMP_DB_PASSWORD server/.env

# 5. ★ 关键：重建计数器（见上方警告）
php server/database/fix_cdr_counter.php

# 6. 数据库结构（本版无 schema 变更，init.php 仅保险，可省略）
php server/database/init.php

# 7. Node 依赖（本版 package.json 未变，跳过；若将来变了需重装）
#    cd server/cdr-receiver && npm install --production
#    cd server/ws-gateway && npm install --production

# 8. 重启服务
pm2 restart all
nginx -s reload

# 9. 验证（见第三节）
```

## 二、全新部署（服务器还没有这套系统）

按 `docs/线上部署指南.md` 完整走一遍，到「上传文件」那步用 v17 包即可：

1. 服务器环境：Node ≥ 18、PM2、PHP 8.1 + 扩展（pdo_mysql/openssl/curl/mbstring/hash/ctype/json）
2. 防火墙放行 **UDP 5140**（话单，最易漏）、TCP 80/443、TCP 8089（WS）
3. 建库 `yump`（utf8mb4），用户 `yump`
4. 上传并解压 v17 包到 `/www/wwwroot/yump`
5. `cp server/.env.example server/.env` 并填 4 个密钥；`/etc/yump/.env.node` 填 DB 密码 + WS 密钥（须与 .env 一致）
6. `php server/database/init.php` 建表
7. **`php server/database/fix_cdr_counter.php`**（空库也会跑，无害）
8. PM2 启动 cdr-receiver + ws-gateway；cron 配心跳
9. Nginx：server 级 `root /www/wwwroot/yump`，location / 指向 `dist/`
10. 权限 `chown -R www:www`，安全清理调试文件（见部署指南九）

## 三、上线验证清单（最少必做）

```bash
# 1. API 通
curl -s http://127.0.0.1/api/nodes        # 返回 JSON（可能空数组）

# 2. 登录
curl -s -X POST http://127.0.0.1/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"yump"}'     # 返回 token

# 3. CDR 计数器已重建（应与各日表行数一致）
php server/database/fix_cdr_counter.php   # 再跑一次应报告无变化/秒级完成

# 4. CDR 列表无筛选查今天（应秒回，total 正确，前端显示真实总数）
curl -s "http://127.0.0.1/api/cdr?pageSize=50" \
  -H "Authorization: Bearer <token>" | head -c 200

# 5. ★ v17 新增：深翻页也秒回（造 10 万条后翻到最后一页验证）
curl -s "http://127.0.0.1/api/cdr?pageSize=50&page=2000" \
  -H "Authorization: Bearer <token>" | python3 -c "import sys,json,time;print('total=',json.load(sys.stdin)['data'] and 'ok')"

# 6. 进程与端口
pm2 status            # cdr-receiver + ws-gateway 均 online
ss -uln | grep 5140   # 有监听
ss -tln | grep 8089   # 有监听

# 7. 前端可访问：浏览器打开域名 → 登录页 → 通话记录 → 翻到任意页应无延迟
```

完整 15 项清单见 `docs/线上部署指南.md` 第十节。

## 五、v17 包内容说明

- 文件：`vos3000-deploy-20260713-v17.tar.gz`（约 1 MB）
- 含：`dist/`（前端构建产物）+ `server/`（PHP 后端 + Node 服务 + fix_cdr_counter.php）+ 部署配置（nginx.conf / ecosystem.config.js / index.html / 404.html）+ `docs/`
- 已排除（不进包）：`server/.env`（密码）、`server/logs/`、`node_modules`、`pm2-dev.json`（含明文密码/本地路径）
- 相对 v16 的增量：**CDR 无筛选分页改用 counter 分段定位**（任意页码毫秒级 + 真实 total + 分页按总条数铺开），前端分页条显示真实总数
- 性能实测（10 万条多天数据，本地）：第 1 页 0.15ms / 中间页 6.4ms / 最后页 7.7ms，旧 UNION 全表方式约 43ms，加速 5.7~281 倍

## 六、关于 Redis（本版未引入，说明决策）

v17 **没有**引入 Redis。原因：当前「秒查总数 + 毫秒级深翻页」的诉求已用 `y_cdr_counter` 预聚合表 + 分段定位解决，counter 表本质上是「穷人版缓存」，覆盖了 Redis 在这里能提供的 90% 价值。

2C4G 阿里云实例若要加 Redis：能跑（Redis 内存可控在 256~512MB、CPU 极轻），但当前单实例内部中控平台流量小，加 Redis 只会增加运维负担与内存压力，收益有限。建议先上 v17 观察监控（QPS / 内存），真出现横向扩展或多实例共享会话需求时再引入。
