# CDR 通话记录查询性能优化报告（2026-07-13）

## 一、问题背景

用户反馈：通话记录页面"没有过滤条件的情况下查询太慢，像是扫描了全表"。诉求是像业界系统那样"几百万数据一秒查出来"，思路是"用 LIMIT 查最新数据 + 快速获取总数"。

## 二、架构诊断

当前 CDR 查询架构为**按天分表 + 预聚合计数**：

- **存储层**：话单按天分表 `y_cdr_YYYYMMDD`，每张表 `id` 主键、`received_at` 有索引（`idx_received_at` 6 张日表均齐全）
- **预聚合层**：`y_cdr_counter(date, node_id, total)` 由 cdr-receiver 收到话单时增量 `total += 真实新增条数`
- **查询层**：`handleList` 按日期范围路由到日表，分两条路径

```
前端请求（默认查"今天"）→ cdrDayTables() 路由日表
  ├─ 单天 && 无筛选 → counter 快路径（total=SUM(counter)，数据=id DESC LIMIT）✅ 快
  └─ 多天 或 带筛选 → else 分支（total=逐表 COUNT(*) SUM = 全表扫描；数据=UNION 物化全表排序）❌ 慢
```

### 慢点根因（两个）

1. **total 用精确 COUNT(*) 扫所有日表**：上一轮为修 EXPLAIN 估算不准的 bug，把跨天/带筛选的 total 改成精确 COUNT(*)。估算是快但错，现在是准但慢——用户体感的"全表扫描"即来源于此。
2. **数据查询 UNION 物化全表**：`SELECT * FROM (UNION ALL 所有日表) t ORDER BY received_at DESC LIMIT`，派生表上 `received_at` 索引完全失效，MySQL 必须把所有表全部行物化进临时表再排序取前 N 条，数据量越大越慢。

## 三、关键前置发现：counter 与日表严重不一致

动手前验证方案 A 的前提（"用 counter 算总数"可信吗），结果 **counter 漂移严重**：

| 日表 | 实际行数 | counter | 状态 |
|------|---------|---------|------|
| y_cdr_20260706 | 3 | 0 | ❌ MISMATCH |
| y_cdr_20260707 | 11 | 0 | ❌ MISMATCH |
| y_cdr_20260711 | 1 | 3008 | ❌ MISMATCH |
| y_cdr_20260712 | 0 | 0 | OK |
| y_cdr_20260713 | 0 | 0 | OK |
| y_cdr_20260714 | 0 | 0 | OK |

根因：cdr-receiver 增量累加 counter，但日表被 TRUNCATE（之前测试清空）时 counter 未同步清 → 漂移。若直接上方案 A 会变成"快但错"（07-11 显示 3008 实际仅 1）。

## 四、修复方案（方案 A）

### 4.1 修复 counter 一致性
新增 `server/database/fix_cdr_counter.php`：先清空旧 counter，再按**表名日期**（与 cdr-receiver.updateCounter 口径一致，而非 `DATE(start_time)`）从各日表重新聚合。重建后验证全部一致 ✅。

### 4.2 改 handleList 查询逻辑
- **无字段筛选**（仅日期范围/默认今天）→ `total = SUM(y_cdr_counter.total WHERE date IN 范围天数)`，O(天数) 毫秒级；多表数据查询改为每表 `ORDER BY received_at DESC LIMIT cap` 再 UNION 取 top N（走 `idx_received_at`，只物化 pageSize 量级的行而非全表）。
- **带字段筛选**（caller/callee/gateway/account/duration/disconnect_cause/cdr_id）→ counter 无法预聚合，仍精确 COUNT(*) 逐表 SUM。

### 4.3 改 handleClear
清空日表时同步 `DELETE y_cdr_counter WHERE date=?`，从根因上防止未来漂移。

### 4.4 修复过程中踩的两个 SQL 坑
- **UNION 分支 ORDER BY/LIMIT 必须括号包裹**：否则 MySQL 报 `1221 Incorrect usage of UNION and ORDER BY`。
- **多表 $multiParams 构造必须先 `$multiParams = []`**：错误写法 `$multiParams = $params; for(...) merge(...)` 会多算一份初始值 → `HY093 Invalid parameter number`。

## 五、验证结果

### 正确性（交叉验证）
- `tests/cdr-dimension-test.php`（21 用例，每例用数据库精确 COUNT 交叉验证 API total）：**21 PASS / 0 FAIL**
- `tests/api-smoke.mjs`：**44 PASS / 0 FAIL**
- `php -l server/api/cdr/index.php`：无语法错误

### 性能（压测，5 万行压测表 y_cdr_perf）
| 指标 | 旧逻辑 | 方案 A | 提速 |
|------|--------|--------|------|
| 数据查询（取最新 50 条） | 物化全表 0.141s | 每表 LIMIT 0.001s | **244x** |
| total 计算 | COUNT(*) 0.004s | counter SUM 0.000s | **18.6x** |
| 跨天 150 万行 total | ≈0.11s（30 表各 COUNT） | ≈0.000s（counter 30 行） | — |

**核心结论**：旧方案耗时 O(数据量) 线性增长（500 万行时物化全表可能十几秒）；新方案耗时 O(天数)+O(pageSize) 恒定快——这正是"几百万一秒查出来"的原理。

## 七、后续建议
1. **counter 一致性监控**：可加定时任务对比 counter SUM 与日表 COUNT，发现漂移自动告警/重建。
2. **带字段筛选场景**：caller/callee 等 LIKE '%x%' 无法走索引，超大数据量下仍慢。如需进一步优化，可引入 Elasticsearch/ClickHouse 做分析存储（当前数据量不必）。
3. **node_id 全为 0**：CDR 数据未正确关联节点（matchNodeId 问题），导致"按节点筛选"实际无效。属数据侧问题，建议另查 cdr-receiver。
4. **半天范围精度**：counter 按天聚合，若用户选"半天"范围（非整天边界），total 会按整天估算略多。当前前端默认整天范围，影响极小；如需精确可在非整天范围时回落到精确 COUNT。
