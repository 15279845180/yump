# yump 多节点管理平台 — 线上部署指南

> 适用环境：CentOS 7.9 + 宝塔面板（Nginx + PHP 8.1 + MySQL 5.7/8.0）  
> 更新日期：2026-07-08

---

## 系统最低要求

| 资源 | 最低 | 推荐 | 说明 |
|------|:--:|:--:|------|
| **CPU** | 1 核 | 2 核+ | 2 个 Node 进程 + PHP-FPM，单核够用但并发高了会卡 |
| **内存** | 1 GB | 2 GB+ | 3 个进程各约 30MB 常驻，PHP-FPM 按需分配；剩余给 MySQL 缓存 |
| **磁盘** | 10 GB | 20 GB+ | 项目本身 ~5MB；CDR 话单日分表是主要增长点（每万条约 3-5MB） |
| **系统** | CentOS 7+ / Ubuntu 18+ | — | 推荐 CentOS 7.9 + 宝塔面板 |
| **网络** | 公网 IP 或内网可达 | — | UDP 5140（话单）、TCP 8089（软电话 WebSocket）需可达 |

### 软件依赖

| 组件 | 版本要求 | 用途 |
|------|----------|------|
| PHP | ≥ 8.1 | API 后端（PHP-FPM） |
| PHP 扩展 | `pdo_mysql` `curl` `json` `mbstring` `openssl` `fileinfo` `bcmath` | 数据库/HTTP/加密 |
| MySQL | 5.7 或 8.0 | 数据存储 |
| Node.js | ≥ 16 | cdr-receiver、ws-gateway 两个常驻服务 |
| PM2 | 最新 | Node 进程守护 + 开机自启 |
| Nginx | 最新（宝塔自带） | 反向代理 + 静态文件 |

### 端口清单

| 端口 | 协议 | 方向 | 组件 | 说明 |
|------|------|------|------|------|
| 80/443 | TCP | 入站 | Nginx | Web 访问 |
| 5140 | UDP | 入站 | cdr-receiver | VOS3000 推送话单 |
| 8089 | TCP | 入站 | ws-gateway | 软电话 WebSocket |
| 3306 | TCP | 本地 | MySQL | 数据库（仅 localhost） |

### 内存估算

```
Nginx:         ~20 MB
PHP-FPM:       ~40 MB × 子进程数（宝塔默认 20 个 = ~800 MB，建议调为 5-10 个）
MySQL:         ~300-500 MB（取决于缓冲池配置）
cdr-receiver:  ~30 MB
ws-gateway:    ~30 MB
───────────────
总计推荐:      2 GB（PHP-FPM 调优后可压到 1 GB）
```

> 💡 宝塔默认 PHP-FPM 开了很多子进程，修改 `/www/server/php/81/etc/php-fpm.conf`：  
> `pm.max_children = 5` 和 `pm.start_servers = 2` 即可省 500+ MB 内存。

从本地打包以下内容上传到 `/www/wwwroot/yump/`：


## 二、服务器环境准备

以下步骤在全新 CentOS 7.9 机器上执行。如果宝塔面板已经装过，部分步骤可以跳过。

### 2.1 安装 Node.js

cdr-receiver 和 ws-gateway 是 Node.js 服务，需要 Node ≥ 18。

```bash
# 方式一：NodeSource 官方仓库（推荐）
curl -fsSL https://rpm.nodesource.com/setup_20.x | sudo bash -
yum install -y nodejs

# 验证
node -v   # 应输出 v20.x.x
npm -v
```

> 宝塔面板的「软件商店」里也有 Node 版本管理器，可以在面板里直接装。

### 2.2 安装 PM2

Node 进程守护，崩溃自动重启。

```bash
npm install -g pm2
pm2 -v
```

### 2.3 检查 PHP 扩展

宝塔 PHP 8.1 默认装了大部分，但确认以下没漏：

```bash
php -m | grep -E "pdo_mysql|openssl|curl|mbstring|hash|ctype|json"
```

| 扩展 | 用途 | 缺失后果 |
|------|------|----------|
| `pdo_mysql` | 数据库连接 | 所有 API 500 |
| `openssl` | bcrypt 密码哈希、AES-256-CBC 加密 | 登录/密码修改失败 |
| `curl` | HTTP 请求 yump API | 节点同步/代理全部失败 |
| `mbstring` | 多字节字符串（中文） | 中文显示乱码 |
| `hash` | SHA256（WS token 签发） | 软电话 token 生成失败 |
| `ctype` | 字符类型校验 | ws-token 参数校验失败 |
| `json` | JSON 编解码 | 所有 API 响应异常 |

宝塔补充安装：**软件商店 → PHP 8.1 → 设置 → 安装扩展**，勾选缺失的扩展后重启 PHP。

### 2.4 关闭 SELinux（CentOS 7 必须）

SELinux 会阻止 cdr-receiver 的 UDP 5140 端口绑定和 PHP 的 curl 外连请求。

```bash
# 临时关闭（立即生效，重启后恢复）
setenforce 0

# 永久关闭
sed -i 's/SELINUX=enforcing/SELINUX=disabled/' /etc/selinux/config

# 验证
getenforce   # 应输出 Permissive 或 Disabled
```

### 2.5 配置防火墙

需要开放以下端口：

| 端口 | 协议 | 用途 | 开放范围 |
|------|:--:|------|----------|
| 80 | TCP | Nginx HTTP | 公网 |
| 443 | TCP | Nginx HTTPS（如果配了 SSL） | 公网 |
| 5140 | **UDP** | CDR 话单接收（VOS 节点推送） | 至少 VOS 节点 IP |
| 8089 | TCP | WebSocket 软电话网关 | 仅本机（Nginx 代理用） |
| 9999 | TCP | 如果配置了额外监听端口 | 公网 |

```bash
# firewalld（推荐）
firewall-cmd --permanent --add-port=80/tcp
firewall-cmd --permanent --add-port=5140/udp
firewall-cmd --permanent --add-port=9999/tcp
firewall-cmd --reload
firewall-cmd --list-ports

# 如果用的是云服务器（阿里云/腾讯云），还需要在安全组里放行以上端口
```

> 🔴 **UDP 5140 不放行 = CDR 话单一条都收不到。** 这是最容易漏的一步。

## 四、宝塔站点配置

### 4.1 创建站点

宝塔面板 → 网站 → 添加站点：

| 设置项 | 值 | 说明 |
|--------|-----|------|
| 域名 | `你的域名` 或 IP | 如 `47.93.231.26` |
| 网站目录 | `/www/wwwroot/yump` | 🔴 **必须是项目根**，不能选 `dist/` |
| PHP 版本 | `PHP 8.1` | |
| 数据库 | 不勾选 | 前面已手动创建 |

> 🔴 **为什么网站目录不能选 dist/**：PHP 后端在 `server/api/` 里，API 请求 `/api/auth/login` → `router.php` → PHP-FPM 需要找到 `/www/wwwroot/yump/server/api/router.php`。如果根目录指 dist，就去找 `dist/server/api/...` → 404。

### 4.2 Nginx 配置

宝塔面板 → 网站 → 点击你的站点 → **配置文件**。

#### 🔴 最关键的一步：检查 server 级 root

打开配置文件后，找到 `server {}` 块最上方的 `root` 指令：

```nginx
server {
    listen 9999;
    root /www/wwwroot/yump;     # ← 必须是项目根！不是 dist/
    # ... 其他配置
}
```

> **这是 404 的头号原因**：如果 `root` 指向了 `dist/`，API 请求 `/api/auth/login` → `try_files` 内部重定向到 `/server/api/router.php` → 宝塔的 PHP 处理器继承 server 级 root → 去 `dist/server/api/router.php` 找文件 → 不存在 → **404**。
>
> 正确做法：server 级 `root` 指向项目根 `/www/wwwroot/yump`，只在 `location /` 里覆盖为 `dist/`。

#### 完整配置（直接替换整个 server 块内的 location 部分）

保留宝塔自动生成的 `listen`、`server_name`、`root`、`index`、日志等行，**只替换 location 部分**：

```nginx
server {
    listen 9999;
    server_name 47.93.231.26;
    root /www/wwwroot/yump;              # ← 项目根（不是 dist/）
    index index.html index.php;

    # 宝塔 PHP 配置（保持原位，不要删）
    # include enable-php-81.conf;

    # ═══════════════════════════════════════
    # 1. API 请求 → PHP 后端（必需）
    # ═══════════════════════════════════════
    location /api/ {
        try_files $uri $uri/ /server/api/router.php?$query_string;
    }

    # ═══════════════════════════════════════
    # 2. 确保 router.php 用正确的 root 执行
    #    （解决 server 级 root 指向 dist/ 时 PHP 404 的问题）
    # ═══════════════════════════════════════
    location ~ ^/server/api/.*\.php$ {
        root /www/wwwroot/yump;
        include enable-php-81.conf;
        fastcgi_intercept_errors off;
    }

    # ═══════════════════════════════════════
    # 3. WebSocket 代理 → 软电话网关（必需）
    # ═══════════════════════════════════════
    location /ws/ {
        proxy_pass http://127.0.0.1:8089;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400;
    }

    # ═══════════════════════════════════════
    # 4. 前端 SPA（Vue Router history 模式 fallback）
    # ═══════════════════════════════════════
    location / {
        root /www/wwwroot/yump/dist;
        try_files $uri $uri/ /index.html;
    }

    # ═══════════════════════════════════════
    # 5. 安全加固
    # ═══════════════════════════════════════
    location ~ ^/server/(config|database|utils)/ {
        deny all;
        return 404;
    }
    location ~ /\.env {
        deny all;
        return 404;
    }
    location ~ /\. {
        deny all;
        return 404;
    }
}
```

#### 为什么需要 `location ~ ^/server/api/.*\.php$`

请求流程：

```
浏览器 GET /api/auth/login
  → 匹配 location /api/
  → try_files: /api/auth/login 文件不存在
  → 内部重定向到 /server/api/router.php
  → 重新匹配 location → 命中宝塔的 location ~ \.php$
  → PHP 处理器的 root 继承自 server 级
  → 如果 server root = dist/ → 找 dist/server/api/router.php → 404 ❌
  → 如果 server root = yump/ → 找 yump/server/api/router.php → 200 ✅
```

`location ~ ^/server/api/.*\.php$` 的作用是：**不管 server 级 root 指向哪里**，PHP 路由器始终用 `/www/wwwroot/yump` 作为 root，确保能找到 `router.php`。

> 💡 `enable-php-81.conf` 是宝塔的 PHP-FPM 配置文件，路径通常在 `/www/server/nginx/conf/enable-php-81.conf`。如果你的 PHP 版本不同，改对应数字（如 `enable-php-74.conf`）。

> 💡 如果宝塔配置文件里已经有 `include enable-php-81.conf;`（在 server 级），上面的 `location ~ ^/server/api/.*\.php$` 里的 `include` 可以改成完整的 fastcgi 配置，避免重复 include。简单做法：保留两处 include 不冲突，Nginx 取最精确匹配的 location。

## 五、创建 .env 配置文件

所有 PHP 端环境变量统一用 `server/.env` 文件管理，**不依赖 Nginx fastcgi_param，不依赖 PHP CLI getenv()**。换个环境只需复制一份 `.env` 文件。

### 5.1 从模板创建

```bash
cd /www/wwwroot/yump
cp server/.env.example server/.env
```

### 5.2 编辑 .env 填入实际值

```bash
vi server/.env
```

```ini
# MySQL 数据库密码（必填）
YUMP_DB_PASSWORD=你创建数据库时设的密码

# JWT 签名密钥（必填，64 位 hex，建议用下面命令生成一个）
YUMP_JWT_SECRET=b03db5d0ff61f65649905fce0098a41b682272f26c1c24cd46b53477b85e58f4

# WebSocket 密钥（必填，PM2 的 ws-gateway 也要配同一个值）
YUMP_WS_SECRET=ws_3cdbf523d3e07d91312989fb0ca88fba

# 字段加密密钥（必填，用于加密节点密码等敏感字段，设了就别改！）
YUMP_FIELD_KEY=319086391886db856242de1299824382
```

### 5.3 生成安全的随机密钥

```bash
# JWT 密钥（32 字节 hex）
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# WS 密钥（16 字节 hex）
node -e "console.log('ws_' + require('crypto').randomBytes(16).toString('hex'))"

# 字段加密密钥（16 字节 hex → 32 字符）
node -e "console.log(require('crypto').randomBytes(16).toString('hex'))"
```

> 四个密钥必须各不相同，建议存到密码管理器。

### 5.4 变量对照

| 变量 | 用途 | PHP (server/.env) | Node (.env.node) |
|------|------|:--:|:--:|
| `YUMP_DB_PASSWORD` | MySQL 密码 | ✅ | ✅ |
| `YUMP_JWT_SECRET` | JWT 签名 | ✅ | — |
| `YUMP_WS_SECRET` | WebSocket token | ✅ | ✅ |
| `YUMP_FIELD_KEY` | 字段加密 | ✅ | — |

> 🔴 **为什么不用 Nginx fastcgi_param 了**：PHP CLI 执行 `init.php`、`heartbeat.php` 时不经过 Nginx，`getenv()` 在不同环境的 php.ini `variables_order` 设置下行为不一致。`.env` 文件方式在所有场景下都可靠。

## 六、Node 服务配置（PM2）

> 🔴 **Node 进程不读 `server/.env`**，它通过 `ecosystem.config.js` 从同目录的 `.env.node` 文件读取环境变量。`YUMP_WS_SECRET` 必须和 `server/.env` 里的值一致。

### 6.1 安装 Node 依赖

> ⚠️ `.env.node` 里的 `YUMP_WS_SECRET` 必须和 `server/.env` 里的一致。

## 十一、故障排查

### 登录提示"检查网络连接"

```bash
# 先测 API 是否通
curl -s -X POST http://127.0.0.1:9999/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"yump"}'
```

- **返回 404**：Nginx server 级 `root` 指向了 `dist/`，导致 PHP 找不到 `router.php`。解决方案见 4.2 节，确保：
  - server 级 `root /www/wwwroot/yump;`（不是 dist/）
  - 或者添加 `location ~ ^/server/api/.*\.php$` 块强制指定正确的 root
- **返回 500**：看 Nginx 错误日志 `/www/wwwlogs/域名.error.log`
- **返回 "数据库连接失败"**：检查 `server/.env` 中的 `YUMP_DB_PASSWORD` 是否正确

### CDR 一条都收不到

```bash
# 1. 确认端口在监听
ss -uln | grep 5140

# 2. 确认防火墙放行
firewall-cmd --list-ports | grep 5140

# 3. 确认进程健康
pm2 status | grep cdr-receiver
pm2 logs vos-cdr-receiver --lines 20

# 4. 用 tcpdump 抓包确认是否有数据到达
tcpdump -i any udp port 5140 -n -c 20
```

如果没有 UDP 包到达 → **防火墙 / 安全组没放行**，或者 VOS 节点没配置推送。

### PM2 启动后立即 crash

```bash
pm2 logs --err --lines 50
```

常见原因：
- `YUMP_DB_PASSWORD` 没配 → MySQL 连接失败
- Node 依赖没装 → `Error: Cannot find module 'ws'`
- 端口被占用 → `Error: listen EADDRINUSE`

### 前端页面加载但接口 401

→ `YUMP_JWT_SECRET` 没配或配错了。检查 `server/.env` 文件中的 `YUMP_JWT_SECRET` 是否正确。

### 软电话 WebSocket 连接失败

→ 检查 `server/.env` 和 PM2 `ecosystem.config.js` 两处的 `YUMP_WS_SECRET` 是否**一致**（必须用同一个值）。

### 修改密码失败

→ `openssl` 扩展没开。`php -m | grep openssl` 确认。

### 12.6 CDR 数据清理

CDR 话单按天分表（`y_cdr_YYYYMMDD`），长期运行会积累大量日表。建议定期清理超过 90 天的旧表：

```bash
# 查看所有 CDR 日表
mysql -u yump -p yump -e "SHOW TABLES LIKE 'y_cdr_%';"

# 删除 90 天前的日表（示例：删除 2026年4月的表）
mysql -u yump -p yump -e "DROP TABLE IF EXISTS y_cdr_20260401, y_cdr_20260402;"

# 自动清理脚本（可加入 cron，每月执行一次）
# 放入 /etc/cron.monthly/cleanup-cdr.sh
#!/bin/bash
KEEP_DAYS=90
CUTOFF=$(date -d "$KEEP_DAYS days ago" +%Y%m%d)
mysql -u yump -p密码 yump -e "SHOW TABLES LIKE 'y_cdr_%'" | tail -n +2 | while read table; do
  table_date=$(echo $table | sed 's/y_cdr_//')
  if [ "$table_date" -lt "$CUTOFF" ]; then
    mysql -u yump -p密码 yump -e "DROP TABLE IF EXISTS $table;"
    echo "Dropped $table"
  fi
done
```

## 十三、CDR 性能调优

### 13.1 批量写入机制

cdr-receiver 采用**内存队列攒批 + 批量 INSERT** 模式，而非逐条写入：

```
UDP接收 → CSV解析 → 入内存队列 → 每200ms或200条 → 批量INSERT(多VALUES)
```

相比逐条写入，DB 往返次数减少 **200 倍**（200 条话单合并为 1 条 SQL），显著提升高并发场景下的吞吐量。

### 13.2 可调参数

以下参数通过环境变量配置（在 `/etc/yump/.env.node` 中添加）：

| 环境变量 | 默认值 | 说明 |
|----------|:--:|------|
| `CDR_BATCH_SIZE` | `200` | 攒满多少条触发写入 |
| `CDR_BATCH_INTERVAL` | `200` | 攒批最大等待时间（毫秒） |
| `CDR_QUEUE_MAX` | `10000` | 内存队列上限，超过丢弃最旧话单 |
| `CDR_POOL_SIZE` | `10` | MySQL 连接池大小 |

### 13.3 调优建议

| 场景 | 话单量 | 建议配置 |
|------|--------|----------|
| **小规模**（<100 并发） | <100 条/秒 | 默认参数即可 |
| **中规模**（1000-5000 并发） | 100-1000 条/秒 | 默认参数，无需调整 |
| **大规模**（1万+ 并发） | 1000-10000 条/秒 | `CDR_BATCH_SIZE=500` `CDR_QUEUE_MAX=50000` |
| **极端**（10万并发） | 10000+ 条/秒 | `CDR_BATCH_SIZE=1000` `CDR_QUEUE_MAX=100000` `CDR_POOL_SIZE=20` |

> 💡 批量写入会引入毫秒级延迟（攒批时间），不影响数据正确性。`CDR_BATCH_INTERVAL` 越大，单批写入效率越高但延迟也越大。

### 13.4 内置优化

| 优化项 | 说明 |
|--------|------|
| **节点IP缓存** | `node_ip → node_id` 内存映射，60秒TTL，避免每条话单查DB |
| **成本费率缓存** | `gateway_out → cost_rate` 内存映射，5分钟TTL |
| **按天分表** | 每天一张表（`y_cdr_YYYYMMDD`），避免单表过大 |
| **ON DUPLICATE KEY UPDATE** | 重复话单自动覆盖，无需先查再写 |
| **队列溢出保护** | 超过 `CDR_QUEUE_MAX` 时丢弃最旧话单并告警 |
| **优雅退出** | 收到 SIGTERM/SIGINT 时 flush 剩余队列再退出 |

### 13.5 监控指标

cdr-receiver 每 5 秒输出一次统计摘要：

```
[STATS] 接收=12345 写入=12300 丢弃=0 错误=0 队列=45(峰值200)
```

| 指标 | 含义 | 健康标准 |
|------|------|----------|
| 接收 | UDP 接收总数 | — |
| 写入 | DB 写入成功数 | 应接近接收数 |
| 丢弃 | 队列溢出丢弃数 | **应为 0**；持续 >0 说明 DB 写入跟不上 |
| 错误 | 写入/解析失败数 | **应为 0**；持续 >0 需查日志 |
| 队列 | 当前待写入数量 | 应接近 0；持续高位说明 DB 是瓶颈 |

> 🔴 如果 `丢弃` 持续增长，说明 DB 写入速度跟不上话单接收速度。优先检查 MySQL 性能（慢查询、连接数、IOPS），其次调大 `CDR_QUEUE_MAX` 和 `CDR_POOL_SIZE`。
